# Atlas 80 分加固包记录

日期：2026-06-23

版本：`0.13.0-runtime-80`

## 1. 本轮目标

这轮不是追求“看起来功能很多”，而是把当前版本打到可客观验收的 80 分线：能编译、能烧录、能看版本指纹、能跑 Mac Brain、能统计 OPUS 60ms WebSocket 帧、能解释失败原因，并且文档能指导下一次实机测试。

## 4. 评分口径

`/api/runtime/score` 按 10 个能力项计分：

| 能力 | 权重 |
|---|---:|
| device_firmware | 10 |
| audio_service | 12 |
| opus_uplink | 14 |
| session_runtime | 12 |
| tools | 10 |
| provider_config | 8 |
| web_console | 8 |
| acceptance | 10 |
| ota_manifest | 6 |
| docs | 10 |

当前分不把未配置的 LLM/ASR/TTS 算作通过。ready score 会把“代码已具备但需要真机或 API Key 才能验证”的能力单独标出来，避免自嗨。

## 5. 晚上实机验收顺序

1. 启动 Mac Brain：

```bash
python3 tools/mimiclaw_bridge_macos.py \
  --dualeye-url http://DualEye的IP \
  --host 0.0.0.0 \
  --port 8787
```

2. 打开验收页：

```text
http://Mac局域网IP:8787/acceptance
```

3. 刷入固件后先看：

```text
DualEye /api/status
DualEye /api/selftest
DualEye /api/capabilities
Mac Brain /api/runtime/score
Mac Brain /api/acceptance/report
```

4. 点验收页“OPUS 真流 1.8s”，通过标准：

```text
atlas_frames > 0
sequence_gaps = 0
payload_len_mismatches = 0
speech_segment_count >= 1
DualEye stream.stage 不应停在 task_create_failed / connect_timeout / capture_failed / encode_failed
```

## 6. 仍然没有假装完成的差距

| 差距 | 当前策略 |
|---|---|
| 流式 ASR | 现在只接收 OPUS 帧并统计，下一步把帧送入流式 ASR |
| 流式 TTS 播放 | 当前仍是 TTS WAV 拉流播放，后续再做边合成边播 |
| WakeNet/AEC | 仍是资源探针，不硬上；先确认堆内存、CPU、LCD 刷新和音频稳定性 |
| 真 OTA | 当前是 manifest + package hash，仍需 OTA 分区、回滚和签名策略 |
| 长期稳定性 | 需要晚间真机连续跑 15-30 分钟，记录 runtime session 和错误计数 |

## 7. 对标 xiaozhi 的现实判断

本轮后 Atlas 在“单设备本地桌面机器人原型”的可验收度上可以到 80 分线；但在成熟语音 IoT 栈上仍没超过 xiaozhi。xiaozhi 仍领先于流式 ASR/TTS、AEC/WakeNet、生产 OTA、多设备后台和长期压测。Atlas 当前优势是双眼 IP、电子宠物应用、视觉表达、本地可调试 Brain，以及更适合做桌面陪伴机器人的产品方向。
